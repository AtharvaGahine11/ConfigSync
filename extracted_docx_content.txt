1. Problem Statement

Modern organizations rely on distributed systems and microservice-based architectures where multiple services require access to shared configuration data such as database connection strings, API keys, feature flags, authentication tokens, and service endpoints.

Managing these configurations manually across multiple services introduces several challenges:

Configuration inconsistencies between services

Delayed propagation of updates

Service downtime during configuration changes

Lack of version control and audit history

Difficulty in rolling back incorrect configurations

Single points of failure in centralized systems

The objective of ConfigSync is to design and implement a distributed configuration management platform that provides centralized configuration storage, real-time synchronization, version control, fault tolerance, and configuration rollback capabilities while demonstrating key distributed system concepts such as leader election, replication, heartbeat monitoring, and consensus management.

2. Proposed Solution

ConfigSync provides a centralized platform for storing and managing configuration data across distributed services.

The proposed solution consists of:

A Flask-based backend application

SQLite-based configuration storage

Real-time synchronization using Server-Sent Events (SSE)

Distributed node cluster simulation

Leader election mechanism

Configuration replication

Heartbeat monitoring

Configuration versioning

Point-in-time rollback support

Interactive monitoring dashboard

Whenever an administrator creates or updates a configuration, the leader node coordinates replication across follower nodes. Once a quorum is achieved, the update is committed and propagated to all connected services.

This architecture ensures consistency, fault tolerance, and high availability while allowing services to receive updates without requiring restarts.

7. Implementation Details

Configuration Management

The administrator creates configurations using the dashboard. Example request payload:

{

  "key":   "database_url",

  "value": "postgresql://localhost:5432/app"

}

The configuration is stored in the database and assigned a version number.

Leader Election

The system simulates a Raft-like consensus mechanism with three node states:

LEADER — coordinates all writes and replication

FOLLOWER — replicates from leader; monitors heartbeat

CANDIDATE — initiates election when leader heartbeat is lost

If the leader fails, followers detect the missing heartbeat, transition to CANDIDATE state, and hold a new election to select a replacement leader.

Replication

When a configuration is updated:

Leader receives update.

Database stores revision.

Followers replicate change.

Quorum validates commit.

Update becomes globally available.

Heartbeat Monitoring

Each node periodically updates its heartbeat information. Benefits include:

Failure detection

Cluster health monitoring

Automatic recovery

Rollback

Rollback restores the system to a previous revision. Benefits:

Disaster recovery

Error correction

Audit support

Watch Service

Implemented using Server-Sent Events (SSE). Benefits:

Real-time updates

Low latency synchronization

Event-driven architecture

10. Conclusion

ConfigSync successfully demonstrates the design and implementation of a distributed configuration management service. The system provides centralized configuration storage, real-time synchronization, leader election, replication, fault tolerance, version control, and rollback capabilities.

The project effectively applies distributed systems concepts inspired by Apache ZooKeeper and etcd while providing a practical and interactive platform for understanding modern configuration management architectures. This implementation fulfills the requirements of the System Design case study and serves as a scalable foundation for future enhancements.
