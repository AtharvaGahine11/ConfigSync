Simple Explanation (30 Seconds)

ConfigSync solves the problem of managing application configurations across multiple services in a distributed system.
Instead of manually updating configuration files on every service, ConfigSync provides a centralized platform where configurations are
stored, versioned, synchronized, and automatically propagated to all connected services.
The system also demonstrates distributed systems concepts such as leader election, replication, heartbeat monitoring, fault tolerance,
 and rollback mechanisms.

What Is The Actual Problem?

Imagine a company has:

Service A
Service B
Service C
Service D

All these services use:

DATABASE_URL
API_KEY
JWT_SECRET
FEATURE_FLAGS

Suppose the database URL changes.

Without ConfigSync:

Admin manually updates Service A
Admin manually updates Service B
Admin forgets Service C
Admin updates Service D later

Result:

Service A → New Database
Service B → New Database
Service C → Old Database ❌
Service D → New Database

This is called:

Configuration Drift

Different services have different configurations.

This can cause:

System failures
Downtime
Security issues
Data inconsistency
What Solution Did You Build?

Instead of updating every service individually:

Admin
  │
  ▼
ConfigSync
  │
  ├── Service A
  ├── Service B
  ├── Service C
  └── Service D

Now Admin updates only once:

DATABASE_URL = new_value

ConfigSync automatically:

Stores the change
Creates a new version
Replicates it
Synchronizes all services

Every service receives the update.

No manual work.

Why Is Leader Election Needed?

Professor may ask this.

Suppose there are multiple nodes:

Node A
Node B
Node C

If all nodes try to update data simultaneously:

Conflict

To avoid this:

One node becomes:

LEADER

Others become:

FOLLOWERS

Only the leader can accept writes.

This ensures consistency.

Why Replication?

Suppose leader crashes.

Without replication:

Data Lost

With replication:

Leader
   ↓
Follower 1
Follower 2

Copies exist everywhere.

If leader dies:

Follower becomes new Leader

System continues running.

Why Heartbeats?

Every node sends:

"I'm alive"

every few seconds.

Example:

Leader → Heartbeat
Leader → Heartbeat
Leader → Heartbeat

If heartbeat stops:

Leader Failed

System automatically starts a new election.

Why Rollback?

Suppose Admin accidentally updates:

DATABASE_URL = wrong_value

Application crashes.

Instead of manually fixing:

Rollback to Version 3

System restores previous working configuration.
